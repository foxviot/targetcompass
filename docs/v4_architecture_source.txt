0. 架构目标与技术范围
定义后台研究引擎的执行边界、科学输出和工程约束。
0.1 平台目标
TargetCompass-Discovery 是面向疾病相关分子发现和疫苗候选靶点筛选的后台生信系统。平台接收疾病或研究方向，生成标准化的 DiseaseSpec 与 ResearchSpec，并按固定状态机完成数据检索、分析计划、独立审批、受控执行、结果复核、证据入库、候选评分和报告生成。GPT 承担规划、审批与解释；Codex 只在独立工程闭环中实现或修复受限代码任务；Temporal 与 Nextflow 负责持久化编排和确定性计算；MCP 统一暴露知识、工具和 Prompt。
将疾病或研究方向转换为可验证的 DiseaseSpec、ResearchSpec 和 AnalysisPlan。
由相互独立的 GPT 角色完成疾病标准化、数据审查、方法审批、结果复核和因果证据分级，并在每个阶段通过确定性 Validator。
让下载、预处理、统计和批量分析在模型会话结束后继续运行，并支持恢复、取消、局部重算和完整追溯。
把 bulk、scRNA、GWAS、QTL、共定位、MR、富集、蛋白可及性和安全性分析封装为可注册、可测试、可版本化的模块。
在模块缺失或出现可复现代码故障时，向 Codex 派发最小工程任务包，经过测试、镜像构建和发布门后再进入正式运行。
将每个候选分子的支持证据、反证、Unknown、评分、风险和结论关联到数据、参数、代码、审批和运行日志。
0.2 运行形态
平台采用无界面优先的后台服务架构。Project API 和 Orchestrator 是主要入口；看板只用于查看任务状态、候选排序、失败原因、证据链和报告。所有 GPT 与 Codex 实例均通过对象 ID、MCP Resource 和版本化 Registry 重建上下文，不使用聊天记忆保存项目事实。标准分析只运行已注册模块，动态代码始终进入独立的 Codex 工程环境。

1. 平台定义、目标与边界
把自动化靶点发现拆成可执行、可审查、可追溯和可验收的系统能力。
1.1 平台运行形态
一次标准运行从疾病或研究方向开始。Project API 创建项目；Disease Normalizer 和 Dataset Scout 形成标准疾病范围与可用数据卡；Planner 生成 AnalysisPlan；独立 Reviewer 与确定性 Validator 审批；Work Order Compiler 生成执行任务；Temporal 保存任务状态，Nextflow 运行固定模块；Evidence Database 接收经复核的证据；Scoring Engine 输出候选层级；Report Writer 只引用 evidence_id 生成报告。模型会话结束不会中断任务，任何模型也不能直接访问数据库、文件系统或计算节点。
输入冻结：DiseaseSpec/ResearchSpec 包含疾病定义、比较组、物种、组织/细胞、靶点类型、因果证据要求、统计单位、输出和禁止条件。
计划审批：AnalysisPlan 先由独立 Method Reviewer 检查科学合理性，再由 registry、schema、许可、配额和策略规则做确定性验证。
受控执行：Orchestrator 以不可重复提交的 idempotency key 启动作业，并返回 run_id。
确定性分析：Nextflow 只拉取固定 digest 的容器和模块；网络、CPU、内存、GPU、运行时限均由 profile 约束。
证据交付：每条 EvidenceItem 必须关联数据快照、效应或注释、QC、模块版本、artifact、review_status 和 run_id。
能力扩展：只有缺少数据适配器、分析模块或出现可复现代码故障时才创建 Codex 工程任务；发布新版本后重新验证原计划。

2.4 架构决策记录（ADR）
ADR
选择
不选择
理由
ADR-01
对 GPT 只暴露一个 bioflow-mcp Gateway
让模型直连多个数据库、调度器和 Codex server
统一授权、审计、schema 与工具过滤，减少模型看到的攻击面。
ADR-02
Temporal 负责持久化编排
用 Celery/Redis 拼接审批、重试和长任务状态
需要可恢复状态机、信号、查询、确定性重放和分类型重试。
ADR-03
Nextflow DSL2 负责科学执行
由 GPT 每次生成脚本并直接运行
固定模块、容器、profile 和运行报告更适合可重复生信分析。
ADR-04
PostgreSQL 保存元数据，S3/MinIO 保存大文件
把大矩阵和日志塞入编排历史或模型上下文
避免大 payload、便于 checksum、生命周期和独立授权。
ADR-05
Codex 使用独立工程闭环
生产运行失败后原地改代码并继续
区分数据/QC问题与代码问题；任何修改先测试、构建和注册。
ADR-06
应用级 run_id/task_id 管理长任务
把生产状态依赖 MCP experimental tasks
MCP 任务机制仍处于实验状态；平台状态需稳定、可迁移。
ADR-07
计划与结果均以严格 schema 交换
依赖模型生成的自然语言步骤
自然语言只能解释；执行输入必须可验证、可哈希、可比较。
ADR-08
所有 GPT/Codex 读取同一版本化知识库
依赖各自聊天记忆或临时提示
疾病词典、方法合同、prompt 与评分规则必须具有统一版本和审计。
ADR-09
Evidence Database 是科学事实源
把 GPT 报告或自由文本当作证据
只有可追溯的模块结果和经复核注释才能进入候选评分。
ADR-10
评分由确定性引擎执行
让 GPT 临时设定权重或直接排名
硬门控、权重、缺失处理和稳定性分析必须可复现。
MCP 的位置  MCP 是模型与平台之间的协议边界，不是队列、调度器或工作流引擎。所有会改变状态的工具最终都调用 Orchestrator 或受控内部服务。

3. 固定知识库、核心对象与数据模型
所有 GPT、Codex 和工作流读取同一版本化事实源；对象之间通过稳定 ID 关联，不依赖聊天记忆。
3.1 Fixed Knowledge Base 的组成与发布方式
Fixed Knowledge Base 采用经过 schema 校验、评审、测试和发布的版本化配置集合。每个 Project 在创建时固定 kb_snapshot_id；后续即使词典、方法或评分规则升级，既有运行仍能按原快照重放。知识库只保存规则、合同和目录，具体研究结果进入 Evidence Database。

4. 端到端执行链路与任务状态机
一次运行由 Orchestrator 拆成有明确输入、输出、审批和终止条件的阶段；GPT/Codex 都只是可替换 worker。
4.1 完整任务链路
创建 Project：保存研究问题、访问范围、资源配额、kb_snapshot_id 和目标输出；返回 project_id。
疾病标准化：Disease Normalizer 读取 Disease Dictionary，生成 DiseaseSpec；Validator 检查疾病 ID、同义词、组织/细胞和排除词。
数据发现：Dataset Scout 通过 MCP 调用获准 Source adapter，生成 DatasetCandidate；metadata parser 和 overlap detector 给出 A/B/C/D 资格。
计划生成：Planner 只从 Method Registry 选择模块，生成带依赖、参数、资源估算、审批点和预期制品的 AnalysisPlan。
独立审批：Method Reviewer 检查统计单位、混杂、样本重叠、数据适配和因果分析前提；Deterministic Validator 再检查 schema、registry、许可、配额和策略。
任务编译：Work Order Compiler 把计划拆成 RUN_REGISTERED_MODULE、BUILD_ADAPTER 或 FIX_CODE 三类 work order，并计算 hash。
执行：标准 work order 直接进入 Temporal/Nextflow；只有缺失 adapter/模块或最小 fixture 可复现代码故障时，才进入 Codex 工程分支。
结果复核：Result Reviewer 读取 manifest、QC、日志和结构化输出，返回 ACCEPT、ACCEPT_WITH_FLAGS 或 REJECT；数值结果不由 GPT 重算。
证据与评分：Evidence Ingestor 写入 EvidenceItem，Causal Reviewer 对 GWAS/QTL/coloc/MR 证据分级，Scoring Engine 按固定 rubric 生成 TargetScore。
报告与归档：Report Writer 只引用 evidence_id 和 score_id；Orchestrator 固化报告快照、成本、失败记录和完整 lineage。

4.5 幂等、取消、恢复与局部重算
每个写操作要求 idempotency_key；重复网络请求返回原 object/run，不创建第二份下载、作业或证据。
Dataset stage 的缓存键由 source_id + accession + source_version + content_hash 组成；完成前写临时前缀，checksum 通过后原子发布。
每个 WorkOrder 有独立 attempt_id；Nextflow -resume 的 task hash 和复用列表写入 manifest，不能把旧缓存伪装为新运行。
取消顺序是停止后续调度、取消 executor、标记 partial artifact、保存日志与成本、更新终态；已经 accepted 的历史 EvidenceItem 不覆盖。
只允许从最近的已验证边界局部重算。例如单个 GWAS 路线失败不应重跑 bulk/scRNA；评分规则升级只重建 TargetScore 和 Report，不重跑数值分析。
Temporal Worker 或 API 重启后根据 workflow_id、executor_job_id 和对象存储 checkpoint 恢复；历史升级前运行 replay test。

5.2 角色隔离的具体实现
每个角色使用独立 model_call_id、system prompt、output schema 和 allowed_tools；可以运行在同一 Agent Service，但不共享隐藏会话状态。
Orchestrator 根据阶段构建上下文；模型只看到当前任务需要的对象摘要和 Resource URI，不看到完整项目历史。
Planner 的输出必须由另一独立 Reviewer run 审查；Reviewer 的 prompt 中不包含 Planner 的自由文本推理，只包含 plan JSON 和可追溯输入。
Result Reviewer 读取的是由 Artifact Service 生成的结构化 QC summary；需要看图时使用已登记的 figure artifact，不能凭日志片段猜测。
角色输出先通过 JSON Schema，再通过确定性规则；任何解析失败、额外字段或引用不存在 ID 的输出都视为无效。

疾病标准化和结构抽取可使用较低成本模型；复杂计划、方法审查和因果偏倚复核使用高能力模型；路由由平台配置，不写进用户提示。
模型/Prompt 升级需要在固定 eval 上比较字段召回、非法工具率、反模式识别、因果措辞和成本；未达到门槛不能成为默认版本。
失败时可进行一次 schema 修复调用；仍失败则返回 MODEL_OUTPUT_INVALID，不能把自由文本当计划。

7.5 大 payload、历史与版本升级
Temporal payload 仅保存对象 ID、小型状态和 hash；超过 256 KB 的内容走 claim-check，写入 S3/MinIO 后返回 artifact_id。
数据集列表、EvidenceItem 和评分结果按分页 artifact 保存；Workflow history 不携带矩阵、大表、完整日志或模型原文。
数据集和分析路线可作为 Child Workflow 并行运行，单个路线失败不会丢失其他已完成结果。
长历史在安全阶段边界 continue-as-new；稳定 run_id 与多次 temporal_run_id 的映射保存在 PostgreSQL。
Workflow/Activity 代码升级前对真实历史做 replay test；需要改变分支逻辑时使用显式版本标记。Temporal 的 Durable Execution 依靠事件历史恢复 Workflow。[R6]

9. Codex 受控工程闭环
Codex 只完成受限的软件工程任务；它不选择靶点、不改变科学规则、不跳过 QC，也不直接运行正式研究分析。

spatial
样本级重复；区域/细胞邻域定义可复现

数千个细胞不是数千个生物学重复；缺少 donor/sample ID 时，数据只能作为 C 级定位证据。
整合/批次校正用于注释和可视化，正式条件差异使用原始计数或符合合同的 pseudobulk 输入。

10.5 GWAS、fine-mapping 与 variant-to-gene 路线
表型匹配：记录病例定义、结局、年龄、性别、ancestry、样本量和是否与 QTL/MR 样本重叠。表型不匹配时只能作为相关旁证。
汇总统计校验：统一基因组 build，标准化 variant ID，检查 effect/non-effect allele、频率、beta/OR/SE、P 值和异常值。
locus 定义：按预设 LD/距离或来源的独立 lead variants 形成 loci；保存 LD reference 的 ancestry 和版本。
fine-mapping：优先导入来源可信的 credible set/PIP；需要重算时使用注册模块，并明确单/多 causal variant 假设。
variant-to-gene：合并距离、编码影响、cis-QTL、染色质连接和调控注释；最近基因只是一条证据，不能自动成为目标基因。
输出 gene-level genetic association evidence：locus、credible variants、V2G 方法、支持/冲突、适用 ancestry 和质量。
GWAS EvidenceItem 字段
说明
phenotype_id / definition
标准表型和原始研究定义，避免疾病名称相似但结局不同。
ancestry / n_case / n_control
用于可迁移性和样本量判断。
locus / lead_variant / build
可追溯到具体区域和构建版本。
fine_mapping
credible set、PIP 或“未提供”；缺失不记为 0。
gene_mapping_methods
编码、QTL、距离、染色质等方法及各自证据。
overlap/bias_flags

唯一性基于规范化 evidence payload + source snapshot + method/version + context 计算 hash，防止同一结果重复计分。

Evidence DB 不存大矩阵和完整图像，只存结构化结果与 artifact 引用；大文件由对象存储保存。

11.4 CandidateTarget 构建
Evidence Builder 以稳定 molecular_entity_id 归并跨 modality 证据；gene、protein、isoform 和 peptide 之间用显式 relation 表连接。
一个候选可有多个 route（surface、secreted、ECD、T-cell peptide、B-cell epitope），每个 route 单独执行硬门控和评分。
支持、反证和 Unknown 按 context 分层：物种、组织、细胞、疾病亚型和时间点不同的证据不能静默合并。
候选长名单不设固定 Top N；先保留满足最低疾病相关性且具有可追溯证据的实体，再由 route gates 和评分分层。
“未被发现/研究不足”使用 novelty 和 evidence_density 表达，不创建“undiscovered=true”这样的不可验证字段。

def compile_step(plan: AnalysisPlan, step: PlanStep) -> WorkOrder:    assert plan.status == "APPROVED"    module = registry.get_published(step.module_id, step.module_version)    inputs = lineage.resolve_inputs(plan, step)    contract.validate_inputs(module.input_schema, inputs)    policy.enforce_module(plan.project_id, module, step.parameters)    payload = WorkOrder(        task_type="RUN_REGISTERED_MODULE",        module=module.immutable_ref(),        inputs=inputs.readonly_manifest(),        parameters=module.resolve_parameters(step.parameters),        expected_outputs=module.output_schema,        resources=module.resources,        network=module.network_policy,        retry=module.retry_policy,        trace_id=plan.trace_id,    )    return payload.with_hash(canonical_sha256(payload))async def workflow_submit(req: WorkflowSubmitIn, ctx: ToolContext):    principal = await auth.verify(ctx.access_token, audience="targetcompass-mcp")    await tool_filter.require_visible(principal, ctx.agent_role, "workflow_submit")    schema.validate(req)    plan = await plan_store.get_approved(req.plan_id, req.plan_hash)    decision = await policy.evaluate(principal, "workflow.submit", plan)    await audit.append(ctx.call_id, principal, hash(req), decision, ctx.trace_id)    if not decision.allow:        raise ToolError("POLICY_DENIED", retryable=False)    return await orchestrator.start_or_get(plan, req.idempotency_key, ctx.trace_id)

附录 G. 开发完成检查表
用于代码评审、部署评审和首个疾病项目上线前的最终核对。
检查项
通过标准
Fixed KB
Disease/Source/Method/Prompt/Rubric/Safety 均版本化、可发布快照、Project 固定版本。
Schema/Object IDs
全部核心对象有 schema、稳定 ID、hash、revision 和生命周期约束。
MCP
单一 Gateway；Resources/Tools/Prompts 按角色过滤；写操作幂等；全审计。
Agent
角色隔离、生成者不自审、Structured Outputs、eval 和注入测试达标。
Temporal
状态机、review/human signal、cancel、retry、replay、claim-check 和局部重算已测试。
Work Order
Plan 不能直接执行；任务单固定输入、版本、资源、网络、输出、测试和 hash。
Nextflow
模块合同、容器 digest、profiles、golden/negative tests、QC 和运行报告。
Codex
触发条件、最小任务包、隔离、allowed_paths、CI、SBOM、人工发布和回滚。
Scientific routes
bulk/scRNA/GWAS/QTL/coloc/MR/可及性/安全前提与失败条件明确。
Evidence DB
accepted EvidenceItem 有 source/run/artifact/method/review；反证和 Unknown 保留。
Scoring
硬门控、权重、coverage、安全惩罚、稳定性和 C0-C4 可复现。
Security
OIDC/RBAC/ABAC、egress、secrets、路径、SSRF、镜像、日志脱敏和越权测试。
Operations
SLO/告警、备份恢复、容量/配额、控制平面重启和 runbook。
Report
全部事实引用 evidence_id/score_id；数值一致；因果措辞与限制检查通过。
最终结构  标准路径为“疾病输入 → 多 GPT 规划与独立复核 → MCP 受控访问 → Work Order 编译 → Temporal/Nextflow 确定性分析 → Evidence DB → 固定评分 → 可追溯报告”。Codex 仅在适配器、模块或代码故障分支中工作。
